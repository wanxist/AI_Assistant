## 建议（按优先级）

### 第一优先：LLM 上下文窗口管理

现在 [ChatView.vue:249](f:/AI/workspace/ai-assistant-web/src/views/ChatView.vue#L249) 把 `messages.value` 全部发给 LLM，100+ 轮对话后会直接爆掉 token 限制。

**建议**：发送前只保留最近 N 条消息（比如近 20 轮），或者做滑动窗口按 token 数裁剪。后端 `chat.py` / `chat_stream.py` 收到 `messages` 后也可以做一层保护性截断。

#### ✅ 已实施（2026-05-27）

采用 **token 数为主 + 轮数为硬上限** 的双重裁剪策略，在后端做保护性截断：

| 文件 | 改动 |
|------|------|
| `.env` | 新增 `CHAT_MAX_ROUNDS=30`、`CHAT_CONTEXT_TOKENS=8000`、`CHAT_PAGE_SIZE=20` |
| `src/config.py` | 新增 `chat_max_rounds`、`chat_context_tokens`、`chat_page_size` 配置项 |
| `src/utils/trim_messages.py` | 新建 `trim_messages()` 工具函数 |
| `src/api/routes/chat.py` | system prompt 注入后、LLM 调用前裁剪 |
| `src/api/routes/chat_stream.py` | SSE 流开始前裁剪 |

**设计要点**：
- system prompt 始终保留在最前面，不受裁剪影响
- 从消息尾部向前累积 token，超预算或超轮数即停止
- 被裁掉的消息记录 debug 日志，方便排查
- 参数通过 `.env` 可运行时调整，无需改代码
- DB 持久化仍保存完整消息（裁剪仅影响发给 LLM 的内容）
- 对前端完全透明，无需修改

### 第二优先：会话消息分页加载

`GET /sessions/{sid}` 在 [sessions.py:69](f:/AI/workspace/AI_Assistant/src/api/routes/sessions.py#L69) 用 `.fetchall()` 一次性加载全部消息。长会话会导致慢查询 + 大 JSON 响应 + 前端卡顿。

**建议**：加 `limit` / `offset`（或游标）参数，默认只返回最近 20 条；前端滚动到顶时加载更早的消息。

#### ✅ 已实施（2026-05-28）

采用 **游标分页**（`before_id`）而非 offset，新消息插入不会导致漏看或重复：

| 文件 | 改动 |
|------|------|
| `src/api/schemas.py` | `SessionDetail` 加 `has_more: bool = False`，消息中包含 `id` 字段 |
| `src/api/routes/sessions.py` | `GET /sessions/{sid}` 加 `limit`(默认20) / `before_id` 参数；SQL 改为 DESC+LIMIT 子查询翻转；多查 1 条判断 `has_more` |
| `src/views/ChatView.vue` | `switchTo()` 传 `?limit=20`，存 `hasMore`；新增 `loadMore()` 取最早消息 id 做游标向前加载；模板顶部加"加载更早的消息"按钮 |

**设计要点**：
- 游标用消息自增 `id`，天然有序，不受新消息插入影响
- `LIMIT + 1` 技巧判断是否还有更多，无需额外 COUNT 查询
- 子查询 `ORDER BY id DESC LIMIT` 再外层 `ORDER BY id ASC`，始终返回时间正序
- `before_id` 为 `None` 时查最新 N 条；有值时查指定 id 之前的 N 条
- 前端 `loadMore()` 把旧消息拼到数组头部，保持时间正序

### 第三优先：存储层优化（摘要化 + DB 改造）

当单会话消息数超过阈值（如 200 条），对早期消息生成摘要，用摘要替代原始内容。这能解决**存储增长**和**LLM 上下文**两个问题。

DB 层配合改造，一次 migration 到位：

| 阶段 | 内容 |
|------|------|
| **Phase 1 — Schema** | 加索引 `(session_id, id)` 加速分页查询；`t_session_message` 加 `summarized` 标记列，为摘要化预留 |
| **Phase 2 — 摘要化** | 触发阈值判断 → 调用 LLM 生成摘要 → 标记已摘要的消息；保留策略：超过 N 天且已摘要的消息可归档清理 |

**摘要化的效果**：旧消息不丢弃，而是压缩为一段脉络描述发给 LLM，让模型仍然知道"之前聊过什么"，同时 token 占用可控。

#### ✅ 已实施（2026-05-28）

一次 migration 覆盖 Schema + 摘要化，9 个文件改动：

| 文件 | 改动 |
|------|------|
| `scripts/V20260528.01__storage_optimization.sql` | 新建迁移：`t_session_info` 加 `summary` 列；`t_session_message` 加 `summarized` 标记列；建 `(session_id, id)` 复合索引 |
| `.env` | 新增 `CHAT_SUMMARIZE_TRIGGER=200`、`CHAT_SUMMARIZE_KEEP_RECENT=20` |
| `src/config.py` | 新增 `chat_summarize_trigger`、`chat_summarize_keep_recent` 配置项 |
| `src/utils/summarizer.py` | 新建摘要服务：`get_summary()` 读摘要、`summarize_session()` 增量摘要 + LLM fallback |
| `src/api/routes/chat.py` | LLM 前注入摘要到 system prompt；保存后检测触发摘要 |
| `src/api/routes/chat_stream.py` | LLM 前注入摘要（无 system prompt 时自动创建） |
| `src/api/routes/sessions.py` | `GET /{sid}` 返回 `summary` 字段 |
| `src/api/schemas.py` | `SessionDetail` 加 `summary: str` |
| `src/views/ChatView.vue` | 消息列表顶部新增摘要卡片（浅蓝背景，最大高度 120px） |

**设计要点**：
- 增量合并：已有摘要 + 新消息 → LLM 生成更新版摘要
- LLM 失败回退：取前 500 字符简单拼接，不影响主流程
- system prompt 注入，`trim_messages` 无需改动
- `summarize_session()` 独立 DB 连接，不影响请求响应
- 2026-05-28 迁移已执行，Schema 验证通过
