# AI_Assistant RAG 系统深度评价报告

---

## 功能模块一：文档摄取与分块（Ingestion & Chunking）

**评分：3/10** — 存在架构级缺陷

### 详细分析

**流程现状：**
```
文件 → Chunker(1024/100, sentence策略) → ParsedDocument块
     → SentenceSplitter(512/50) → 节点
     → 嵌入(原文) → jieba分词(覆写) → 向量库
```

### 优点
- 多格式支持（PDF/图片/DOCX/PPTX），自动扫描件检测走 OCR
- MD5 去重避免重复上传
- 嵌入基于原始文本计算（语义保真），BM25 用分词文本（检索优化），思路正确

### 缺点

1. **双重分块（P0 缺陷）**：`upload.py:74` 用 `Chunker(chunk_size=1024, overlap=100)` 切一次，`ingestion.py:31` 用 `SentenceSplitter(chunk_size=512, overlap=50)` 再切一次。第二重完全不感知第一重的边界，语义完整性被破坏。同时报告给用户的 `chunks_count` 是第一重的数量而非实际入库数，指标失真。

2. **分词覆写原文（P0 缺陷）**：`ingestion.py:56-58` 用 jieba 分词结果覆写 `node.set_content()`。嵌入向量基于原文、存储文本是分词版、LLM 收到的也是分词版（带空格的中文）。三个环节看到的文本不一致。重排器收到的是"根据 我们 的 分析 结果 显示"这种带空格的异常文本，影响打分精度。

3. **流式与非流式上传参数不一致**：阻塞式用 `chunk_size=1024`、流式用 `chunk_size=512`，且流式不传 `extra_metadata`（缺少 `pages` 字段）。同一文件走不同入口结果不同。

4. **摘要生成失败完全静默**：`upload.py:90-91` 的 `except Exception: pass` 连一行日志都没有。

5. **`status="indexed"` 判断不准确**：基于 `chunks` 是否为空判断，不验证向量库是否真正写入成功。

## 功能模块二：向量存储（Vector Store）

**评分：5/10** — 功能可用但细节粗糙

### 优点
- pgvector + ChromaDB 双模，启动时自动选可达的
- 单例 + `@lru_cache`，不反复建连
- jieba 分词 + PG `to_tsvector('simple')` 方案正确，避免安装 zhparser 扩展

### 缺点

1. **`embed_dim=1024` 硬编码**（`index_store.py:59`）：不读 `settings.embedding_dim`。用户改了维度，模式与向量不匹配，插入静默失败。

2. **静态回退**：启动时选一次，运行中 PG 宕机不会切 ChromaDB。`@lru_cache(maxsize=1)` 导致选择在进程生命周期内不变。

3. **无连接池**：每次操作走 `psycopg.connect()` 新建连接。100 并发查询 = 100+ PG 连接。

4. **`text_search_config="simple"` 写死**、`table_name="documents"` 写死，不可配置。

5. **未创建 GIN 索引**：大量文档时 BM25 查询缺少索引会线性扫描。

6. **连接未在 finally 中关闭**：`_check_pgvector()` 和 `_init_pgvector_schema()` 中，异常时连接泄漏。

## 功能模块三：分词 & BM25（Tokenizer）

**评分：7/10** — 方案正确，细节待完善

### 优点
- jieba 应用层分词 + `text_search_config="simple"` 方案成熟
- 只对含 CJK 字符的文本分词，英文透传
- 查询侧和存储侧都分词，一致性好
- jieba 不可用时静默降级

### 缺点

1. **覆写原文问题**：分词后的空格文本被存储为节点内容（如前述）

2. **CJK 检测不完整**：正则 `[一-鿿㐀-䶿]` 不覆盖日文假名、韩文，日语/韩语文档跳过分词

3. **无自定义词典**：无法添加领域术语（产品名、专业词汇），分词精度受限于 jieba 默认词典

4. **无 BM25 分数归一化**：混合搜索时 BM25 分数和向量相似度不在同一尺度

---

## 功能模块四：混合检索（Hybrid Retrieval）

**评分：5/10** — 功能存在但实现粗糙

### 优点
- 向量 + BM25 混合搜索，设计方向正确
- 短查询自动加权（qlen<15 → coarse_k+10），BM25 对关键词查询更有效
- 两阶段粗精排设计合理

### 缺点

1. **`retrieve()` 和 `retrieve_with_rerank()` 代码重复严重**：前 40 行几乎完全相同（embedding、tokenize、VectorStoreQuery、score 绑定），约 25 行重复代码。

2. **`object.__setattr__(node, 'score', score)` 脆弱**：绕过 LlamaIndex 的属性管理机制直接注入属性，升级可能崩溃。

3. **混合搜索非真正融合**：`mode="hybrid"` 实际是两条独立 SQL 的 `UNION` 去重，dense 优先，sparse 分数被丢弃。不是 alpha 加权融合。

4. **`retrieve_with_rerank()` 中文本到节点映射会丢节点**：`text_to_node = {node.get_content(): node for node in coarse_nodes}`。如果两个节点文本相同，后者覆盖前者，早期节点静默丢失。

5. **ChromaDB 回退时混合模式可能失效**：ChromaDB 不支持 `mode="hybrid"`，静默降级为纯向量搜索，无日志提示。

6. **查询 embedding 无缓存**：同一问题重复查询每次都调 Embedding API。

---

## 功能模块五：精排（Reranker）

**评分：8/10** — 最好的模块

### 优点
- `bge-reranker-large` 选用正确，中文语义匹配效果好
- FP16 加速，单例加载
- 先本地缓存再 HuggingFace，离线可用
- `compute_score` 单/多返回值做了兼容
- 批处理调用

### 缺点

1. **模型路径依赖 `__file__.rsplit("src", 1)[0]`**：部署时改变目录结构就挂

2. **无 batch 内归一化**：不同 batch 间分数不可比

3. **收到的是分词文本而非原始文本**：如前所述

4. **首次加载 2GB 模型无预热**：第一次调用要等几十秒

---

## 功能模块六：查询引擎（Query Engine）

**评分：4/10** — 问题最集中的模块

### 优点
- 两阶段检索（直达 → HyDE 深度搜索）设计方向正确
- HyDE 失败不阻断流程
- 来源引用格式清晰（[1] (文件名)）

### 缺点

1. **0.35 阈值是魔法数字（P0）**：出现在两处（`query_engine.py:85,95`），两次比较的是不同分数体系的分数。Stage 1 是余弦相似度，Stage 2 是 cross-encoder 分，却用同一个 0.35。cos=0.35 和 reranker=0.35 含义完全不同。

2. **阈值同时用于两个不同目的**：第 85 行用作"进入 Stage 2 的门槛"，第 95 行用作"最终置信度门槛"。当 Stage 1 分数 > 0.35 时不触发 Stage 2，但 95 行的检查仍无意义地再执行一遍（肯定通过）。

3. **`doc_ids` 过滤是死参数**：schema 接受，但 `query_engine.py` 不传给 retriever，retriever 也没有过滤能力。传了等于没传。

4. **上下文无 token 限制**：所有节点 `"\n\n"` 拼接，5 个 1024 字的节点 = 5000+ token，可能溢出 LLM 上下文窗口被截断。

5. **HyDE prompt 设计有问题**："不需要真实准确，只需要写出一个看起来像答案的段落"——让 LLM 故意编造，这在事实性 RAG 场景可能引入噪声。

6. **`_do_retrieve()` 异常捕获不完整**：只捕获 `ImportError/ModuleNotFoundError`，任何运行时异常（超时、连接断开）直接 500。

7. **异步端点调用同步方法**：`query.py` 的 `async def` 里调 `engine.query()`，阻塞事件循环。

8. **prompt 模板加载失败无兜底**：`load_prompt` 返回空字符串时直接传给 LLM。

9. **HyDE 失败零日志**：`except Exception: return None`

10. **15+ 参数硬编码**：COARSE_TOP_K、FINE_TOP_K、两个 0.35、HyDE temperature、max_tokens 等都不能通过 `.env` 配置。

---

## 功能模块七：Prompt 工程

**评分：6/10** — 基础可用，缺精细度

### 优点
- YAML 分离，非技术人员可调
- 要求编号引用、禁止编造、不足坦言
- 中英双语模板

### 缺点

1. **无文档元信息注入**：不传上传时间、文档类型给 LLM，无法判断时效性
2. **无 few-shot 示例**：边界输出格式不稳定
3. **`system.yaml` 休眠 bug**：值是 dict 但 loader 当 string 处理
4. **无分文档类型的差异化 prompt**
5. **无 prompt 注入防御**：用户问题通过 `+` 直接拼入 HyDE prompt（`query_engine.py:134-136`），文档内容无过滤注入 LLM 上下文。恶意文档可操控 LLM 输出。

---

## 功能模块八：前端 RAG 集成

**评分：4/10** — 核心流程通，细节缺失严重

### 优点
- RAG 优先于 LLM 的设计（先 `/query` → 不行再 `/chat/stream`）
- 有拒答关键词检测

### 缺点

1. **RAG 来源完全丢弃**：后端返回 `sources`（doc_id、filename、score、snippet），前端 `ChatView.vue` 只取 `answer`，来源信息完全不展示

2. **拒答判断用前端硬编码关键词列表**：`['无法完整回答', '无法基于现有资料', '没有找到', ...]`。后端 prompt 一改，前端关键词可能全失效。且只有中文关键词，英文拒答无法识别

3. **`top_k=3` 硬编码在前端**：无法调整

4. **无流式输出**：长回答等全部生成完才显示

5. **只有 `/sessions` 做了鉴权**：`/query`、`/upload`、`/chat`、`/documents` 全公开

---

## 功能模块九：可观测性（Observability）

**评分：2/10** — 几乎为零

### 问题

1. **低分查询零追踪**：score ≤ 0.35 被拒答的查询不记录。无法分析知识库覆盖缺口，无法科学调阈值
2. **无检索质量评估**：无 eval 数据集，无 precision/recall 指标
3. **无 latency 打点**：不知道慢查询卡在 embedding / 检索 / 重排 / LLM 哪个阶段
4. **HyDE 失败静默**、摘要失败静默
5. **每次查询不记录 coarse 候选数、reranker 分数分布**
6. **无 RAG 核心逻辑的功能测试**：21 个测试中，`retrieval.py`、`ingestion.py`、`query_engine.py`、`tokenizer.py`、`chunker.py` 完全零测试覆盖。已有测试只是烟雾测试（检查返回 200、JSON 有 answer 字段）

---

## 功能模块十：安全（Security）

**评分：3/10** — 多个严重漏洞

1. **`/query`、`/upload`、`/chat`、`/documents` 无鉴权**
2. **JWT secret 硬编码**：`auth.py:17` 的 `getattr(settings, 'jwt_secret', 'change-me-in-production')`，config.py 没有 `jwt_secret` 字段，所有 token 用字面量 `'change-me-in-production'` 签名
3. **prompt 注入无防御**：用户问题拼入 HyDE prompt，文档内容注入 LLM 上下文
4. **上传无文件类型白名单、无大小限制**
5. **`.env` 文件含真实 API key 且在仓库中**
