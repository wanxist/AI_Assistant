## P0: doc_ids 过滤失效

### 现象

API 请求 `POST /query` 或 `POST /query/stream` 传入 `doc_ids` 参数后，过滤完全无效——返回的是全库检索结果。

### 根因：参数在每一层被静默丢弃

| 层级 | 文件 | 行号 | 问题 |
|------|------|------|------|
| Schema | `src/api/schemas.py` | 66 | `doc_ids: list[str] \| None = None` 正确定义 |
| Route | `src/api/routes/query.py` | 24, 38 | `req.doc_ids` **未传递**给 `engine.query()` / `engine.query_stream()` |
| Engine | `src/knowledge/query_engine.py` | 50, 111, 150 | `query()`, `query_stream()`, `_retrieve()` 都没有 `doc_ids` 参数 |
| Engine | `src/knowledge/query_engine.py` | 35-37 | `_cache_key()` 不含 `doc_ids`，不同 doc_ids 的查询会缓存碰撞 |
| Retriever | `src/knowledge/retrieval.py` | 34, 55-59, 76, 97-101 | `retrieve()` / `retrieve_with_rerank()` 没有 `doc_ids` 参数；`VectorStoreQuery` 不设置 `filters` |

### 底层能力（已具备）

- pgvector 的 `query()` 通过 `MetadataFilters` 支持 SQL 级过滤：`metadata_->>'doc_id' IN ('...', '...')`
- ChromaDB 同样支持 `MetadataFilters` → `where` 子句
- `ingestion.py:38` 已将 `doc_id` 存入每个 `TextNode` 的 metadata

### 修改方案

```
route (query.py):     传递 req.doc_ids → engine.query(question, top_k, doc_ids)
engine (query_engine): 新增 doc_ids 参数 → 传递给 _retrieve → _do_retrieve → retriever
                       同时放入 _cache_key 防止缓存碰撞
retriever (retrieval): 新增 doc_ids 参数 → 构建 MetadataFilters(FilterOperator.IN, FilterCondition.OR)
                       → 设置 VectorStoreQuery.filters
```

## P1: 评分阈值混用（两套不兼容的体系）✅ 已修复

### 问题概述

RAG 检索有两个阶段，产生的分数含义完全不同，但代码把两类分数混在一起控制：

| | Stage 1 (余弦相似度) | Stage 2 (Reranker) |
|---|---|---|
| 来源 | pgvector cosine distance→similarity | bge-reranker cross-encoder logits → min-max归一化 |
| 范围 | [0, 1]（绝对值） | 原始: (-∞, +∞); 归一化后: [0, 1]（批次相对值） |
| 稳定性 | 稳定 | 归一化后不稳定（取决于同批次其他候选项） |

三个具体 Bug：

1. **死代码** — `retrieval_stage2_threshold = 0.0` 检查在归一化后的 [0,1] 分数上永远不触发
2. **过滤顺序错误** — `min_score` 在归一化后过滤（批次相对分），导致垃圾通过、好结果被丢弃
3. **同一阈值用于不同量纲** — `0.35`（余弦）和 `0.1`（归一化后）语义完全不同，却放在同一个配置组里

### 修复内容

#### 1. `src/knowledge/reranker.py` — 核心修复

**问题：** `min_score` 过滤在 min-max 归一化（第 84-88 行）**之后**执行（第 90-95 行），导致过滤条件 `s >= 0.1` 操作在批次相对分数上。

**修复：** 把过滤移到归一化**之前**，操作在原始 cross-encoder logits 上。

修改前流程：
```
计算原始 logit → 归一化 [0,1] → min_score 过滤 → 返回
```

修改后流程：
```
计算原始 logit → min_score 过滤（原始 logit）→ 归一化 [0,1] → 返回
```

具体改动（`reranker.py:78-98`）：

```python
# 改前
# Score normalization: min-max to [0, 1]
if ranked and ranked[0][1] != ranked[-1][1]:
    min_s = ranked[-1][1]
    max_s = ranked[0][1]
    ranked = [(t, (s - min_s) / (max_s - min_s)) for t, s in ranked]

if min_score:
    kept = []
    for item in ranked:
        if item[1] >= min_score or not kept:
            kept.append(item)
    ranked = kept

# 改后
# Filter on raw logits before normalization
if min_score is not None:
    kept = []
    for item in ranked:
        if item[1] >= min_score or not kept:
            kept.append(item)
    ranked = kept

# Score normalization: min-max to [0, 1] — applied AFTER filtering
if len(ranked) >= 2 and ranked[0][1] != ranked[-1][1]:
    min_s = ranked[-1][1]
    max_s = ranked[0][1]
    ranked = [(t, (s - min_s) / (max_s - min_s)) for t, s in ranked]
```

额外守卫：把 `if ranked and` 改为 `if len(ranked) >= 2 and`，避免过滤后只剩 1 条时做无效归一化。

**为什么保留归一化而不是直接返回原始 logits：** 归一化后的分数会作为 `node.score` 最终存入 `SourceInfo`，前端用这个分数显示"相关度"。如果直接返回原始 logits（可能是 -5 到 +10），用户看到负数或几十的分数会很困惑。[0, 1] 的展示更友好。

#### 2. `src/config.py:65-68` — 调整默认值和注释

```python
# 改前
retrieval_stage2_threshold: float = 0.0  # reranker logits, >0 = relevant
rerank_min_score: float = 0.1  # min-max normalized, filters low-confidence results

# 改后
retrieval_stage2_threshold: float = 0.0  # deprecated — filtering moved into reranker
rerank_min_score: float = 0.0  # raw logit threshold (BGE: >0 = relevant), applied pre-normalization
```

- `rerank_min_score` 从 `0.1` 改为 `0.0`：因为操作对象变成了原始 logit，BGE 模型以 0 为"相关/不相关"分界线
- `retrieval_stage2_threshold` 标记为 deprecated：它的语义已合并到 `rerank_min_score` 中

#### 3. `src/knowledge/query_engine.py:81-82` — 删除死代码

```python
# 删除
if top_nodes and (getattr(top_nodes[0], "score", 0) or 0) <= settings.retrieval_stage2_threshold:
    return {"nodes": [], "sources": [], "context": ""}
```

**为什么删除而不是修复：** reranker 内部已经通过 `min_score` 做了过滤（修复后操作在原始 logit 上），这层检查是冗余的。即使未来有人想在 engine 层再加一层过滤，也应该建一个新的独立阈值配置，而不是复用已经语义混乱的 `retrieval_stage2_threshold`。

#### 4. `src/knowledge/reranker.py:51` — 签名微调

`min_score` 参数类型从 `float | None = None` 保持不变，但语义从"归一化后阈值"变为"原始 logit 阈值"。调用方 `retrieval.py:117` 无需修改，因为它是透传 `settings.rerank_min_score`。

### 修复后效果（以实际场景为例）

**场景：** 用户问"年假有几天？"，知识库 20 条候选，只有 2 条真正关于年假。

| 候选 | 原始 logit | 修复前（归一化后 0.1 过滤） | 修复后（原始 logit 0.0 过滤） |
|------|-----------|--------------------------|---------------------------|
| "5天年假..." | +8.2 → 归一化 1.0 | ✅ 通过 | ✅ 通过 |
| "年假工资..." | +6.1 → 归一化 0.83 | ✅ 通过 | ✅ 通过 |
| "财务报表..." | -3.5 → 归一化 0.35 | ❌ 通过（垃圾） | ✅ 过滤掉 |
| 其余 17 条垃圾 | -2~-4 → 归一化 0.0~0.45 | ❌ 部分通过 | ✅ 全部过滤 |

修复前：LLM 收到夹杂大量不相关内容的上下文，回答质量下降。
修复后：LLM 只收到 2 条精准相关的上下文。

## P1: Query Embedding 无缓存 ✅ 已修复

### 现象

相同问题每次调用 `/query` 都会重新请求 Zhipu embedding API，产生不必要的延迟和费用。

### 当前位置

`src/knowledge/embeddings.py:133-134`:
```python
def encode_query(self, query: str) -> list[float]:
    return self._ensure_model().embed([query])[0]
```

每次调用都走 `_ZhipuAPI.embed()` → `httpx.post()` 请求 Zhipu API。

### 现有基础设施

`src/storage/cache.py` 已有 `MemoryCache`（进程内 TTL 缓存，线程安全），`QueryEngine` 已经在用它缓存 RAG 响应（`query_engine.py:115-116`）。

### 修复内容

`src/knowledge/embeddings.py` — `encode_query()` 加 MemoryCache 层：

修改前（`embeddings.py:133-134`）：
```python
def encode_query(self, query: str) -> list[float]:
    return self._ensure_model().embed([query])[0]
```

修改后：
```python
def encode_query(self, query: str) -> list[float]:
    cache = get_memory_cache()
    key = f"emb:{hashlib.md5(query.encode()).hexdigest()}"
    cached = cache.get(key)
    if cached is not None:
        return cached
    emb = self._ensure_model().embed([query])[0]
    cache.set(key, emb, ttl=3600)
    return emb
```

新增依赖（文件头部）：
```python
import hashlib
from src.storage.cache import get_memory_cache
```

**设计决策：**

- **TTL=3600s（1 小时）**：embedding API 对相同输入返回确定性结果，向量本身不会变化。比 RAG 响应缓存的 300s 长得多，因为 RAG 响应受 LLM 生成随机性和知识库更新影响，而 embedding 不受这些因素影响
- **MD5 哈希键**：embedding 向量本身是 1024 维浮点数列表，不适合直接做 dict key。对原始文本 MD5 得到定长字符串，稳定且碰撞概率可忽略
- **仅缓存 `encode_query`，不缓存 `encode`**：`encode` 是批量接口，输入文本列表通常各不相同（批量索引文档片段），缓存命中率极低。而 `encode_query` 是用户反复输入的相同问题，缓存命中场景明确
- **复用 MemoryCache 单例**：零新增依赖，与 QueryEngine 共享同一套 TTL 清理机制
