# RAG 中的"长文档 vs 大知识库"：功能与技术

这是两个经常被混为一谈但本质不同的问题。分开讲清楚。

---

## 一、先分清两个概念

| 维度 | 长文档（Long Document） | 大知识库（Large Knowledge Base） |
|:--|:--|:--|
| 核心矛盾 | 单文档太长，LLM 上下文窗口装不下全文 | 文档数量太多，检索效率和精度下降 |
| 典型量级 | 单篇 10-200 页 PDF，或 10 万+ token 的文字 | 10 万+ 文档，百万到亿级 chunks |
| 瓶颈在 | 上下文窗口、信息压缩、跨段落推理 | 检索延迟、召回精度、索引膨胀 |
| 解法方向 | 拆分 + 摘要 + 层级理解 | 粗筛 + 精排 + 索引优化 |

---

## 二、长文档：完整理解 vs 上下文限制

### 2.1 拆分策略再审视

之前的分块策略适用于"日常文档"，但面对 100 页+ 的超长文档，需要额外考量：

| 策略 | 适用场景 |
|:--|:--|
| 递归拆分 | 先按章节拆，章节仍超预算再按段落拆，段落还大才按 token 数切 —— 逐级降级 |
| 文档结构感知 + 元数据注入 | 每个 chunk 带 `{doc_title, chapter, page, section}` 元数据，让 LLM 知道这段话的"位置感" |
| 语义保持的 overlap 设计 | 长文档中，overlap 不仅是防止截断，更要保证跨页表格、跨段公式、跨章节引用不丢失 |
| 图像/表格锚定 | 长文档里图表往往承载关键信息，chunk 中保留图表 caption + 描述，并通过 metadata 链接回原图 |

### 2.2 长上下文 LLM 的局限性

即使模型宣称 128K/1M 上下文窗口，直接把整本手册丢进去有致命问题：

| 问题 | 说明 | 结论 |
|:--|:--|:--|
| Lost in the Middle | 上下文中间部分的信息被模型严重忽视，长文档的中间段落召回率断崖下降 | 长上下文不等于长记忆 |
| "大海捞针"不可靠 | 在 128K 上下文中找一句话，模型经常找不到或编造 | 检索仍是必要的 |
| 推理成本 | 每轮对话天价 token 消耗，用户量上来后不可持续 | 成本与规模不可兼得 |
| 注意力稀释 | token 越多，每个 token 分配到的 attention 越少，细粒度信息丢失严重 | 越多反而越"看不清" |
| 正确做法 | 长上下文做"精读"（检索后的精选内容展开），配合检索缩小范围，而非"全文朗读" | — |

### 2.3 长文档关键技术

| 技术 | 作用 | 适用量级 |
|:--|:--|:--|
| 文档摘要链（Map-Reduce / Refine） | Map：每段独立摘要 → Reduce：汇总摘要再摘要，逐段压缩保留层次感 | 单篇 50-500 页 |
| 多粒度索引 | 同时建句子级（精准匹配）+ 段落级（上下文）+ 章节级摘要（全局理解）三种索引 | 任意 |
| RAPTOR | 递归摘要树：叶子→chunk，逐层聚类+摘要，从不同层级检索 | 单篇 100+ 页 |
| LongRAG | 检索单元从 chunk 变大段落（4K+ token），减少 chunk 数量，提升信噪比 | 知识库级 |
| Key Point Extraction | 先提取长文档"关键点"做索引，命中后展开原文 | 技术手册/合同 |

### 2.4 跨文档长推理

用户问的不一定是"这篇文档第 5 页写的什么"，而是"对比 A 文档的质量目标和 B 文档的实施计划，找出不一致"。这是一种 Multi-hop + Long Context 的复合问题：

| 步骤 | 动作 | 产物 |
|:--|:--|:--|
| 1. 检索 | A 文档相关段 + B 文档相关段，分别召回 | 两组候选 chunk |
| 2. 重排序 | Reranker 对两组结果合并精排 | 最相关的 top-N 片段 |
| 3. 生成 | 两段上下文合并 + 原始问题 → LLM 推理 | 对比结论 |

关键不在"把两本手册全文放进去"，而在检索召回是否精准。

## 三、大知识库：检索工程化的深水区

### 3.1 核心矛盾

| 问题 | 原因 | 引发的连锁反应 |
|:--|:--|:--|
| 语义空间拥挤 | 亿级向量在高维空间挤在一起，top-k 可能和问题毫不相关 | Reranker 和元数据过滤的必要性急剧上升 |
| 噪声淹没信号 | 文档越多，和 query 表面相似但不相关的内容越多 | 纯语义检索不可靠，必须混合关键词 |
| 冷热不均衡 | 80% 的问题只涉及 20% 的文档，但每次扫全库 | 热数据缓存 + 分层索引 |
| 更新一致性 | 文档删除/更新后，向量库、BM25、元数据索引需同步 | 增量更新管道 + 版本对齐 |

### 3.2 关键优化技术

| 技术 | 解决什么问题 | 实现方式 |
|:--|:--|:--|
| 多路召回 + 融合排序 | 单路召回遗漏 | 向量 + BM25 + 知识图谱 + 元数据过滤，4 路并行召回 → RRF/学习排序融合 |
| Reranker 精排 | 粗召回精度不够 | `bge-reranker-v2-m3` / `Cohere Rerank v3`：粗召回 top-100 → reranker 精排 → 取 top-5 |
| 查询路由 | 不同问题该搜不同库 | LLM Router 或分类器：技术类→技术文档索引，HR 类→制度文档索引 |
| 查询分解 | 复杂问题拆成子问题 | "对比三个产品的定价" → 拆成 3 个子 query → 分别检索 → 合并结果 |
| 索引分层 | 全量检索太慢 | 粗粒度章节摘要索引（第一层）→ 命中后搜细粒度 chunk 索引（第二层） |
| 预筛选 | 减少检索候选集 | 按时间范围、部门标签、文档类型过滤，候选集从 1000 万缩到 10 万再检索 |
| 缓存热数据 | 高频 query 重复检索 | Redis 缓存热 query 的 top-k，命中后跳过检索直接进生成 |

### 3.3 向量索引的高级选择

| 索引类型 | 适用规模 | 特点 | 精度 vs 延迟 |
|:--|:--|:--|:--|
| HNSW | < 100 万向量 | 图索引，greedy 遍历邻居节点，O(log N) | 高精度 / 低延迟 |
| IVF_FLAT / IVF_SQ8 | 100 万 - 1000 万 | 倒排索引粗筛 + 候选集暴力搜索，可 GPU 加速 | 高精度 / 中延迟 |
| DiskANN | > 1000 万 | Vamana 图 + 量化压缩 + SSD 存储，成本极低 | 中精度 / 低延迟 |
| ScaNN | > 1000 万（Google 方案） | 各向异性量化，同量级召回率显著优于 IVF+PQ | 高精度 / 低延迟 |

以 Milvus 为例：< 100 万用 HNSW，100-1000 万用 IVF_SQ8 + GPU，> 1000 万用 DiskANN 或 ScaNN。

### 3.4 大规模下的增量更新

| 策略 | 方式 | 适用场景 |
|:--|:--|:--|
| 热索引 + 冷索引 | 新增/修改文档进热索引（小，更新频繁），定期 merge 到冷索引（大，只读） | 持续摄入 + 定期检索 |
| 分段索引 | 按时间分段：`idx_2026q1`, `idx_2026q2`，新季度建新索引，过期整段删除 | 按时间分区的归档场景 |
| 双缓冲 | 建新索引期间旧索引继续服务，完成后原子切换 | 大规模重建、零停机需求 |

## 四、两者交叉地带

长文档 + 大知识库叠加时，挑战是乘数关系，不是加法：

| 叠加维度 | 独立时的影响 | 叠加后的放大效应 |
|:--|:--|:--|
| 检索精度 | 大库的向量空间拥挤 | 加长文档 → 更多相似但无关的长文本混入 → 精度断崖下降 |
| 上下文预算 | 长文档需要更多 token | 加大库的 multi-hop 检索 → 每次 hop 携带更多上下文 → 预算快速耗尽 |
| 推理链长度 | 大库需要跨文档 hop | 加长文档 → 每个 hop 携带的内容更多 → 推理链更复杂 |
| Token 成本 | 每次 hop 消耗 tokens | 两因素叠加 → 每次 hop 消耗翻倍 → 成本指数增长 |

**实战推进路线：**

| 优先级 | 动作 | 投入 | 收益 | 为什么先做 |
|:--|:--|:--|:--|:--|
| P0 | 上 Reranker | 极小 | 极大 | 粗召回 100 条 → 精排取 5 条，即插即用 |
| P1 | 元数据过滤 + 预筛选 | 小 | 大 | 把候选集从千万缩到十万，检索压力骤降 |
| P2 | 查询改写/分解 | 中 | 中 | 解决"问得不清楚"导致的召回失败 |
| P3 | 多路召回融合（BM25+向量） | 中 | 大 | 补齐纯语义检索的盲区 |
| P4 | 索引分层 | 大 | 大 | 真正海量时才需要，前期过度设计 |
| P5 | 评估体系（RAGAS） | 小 | — | 没有评估的调优是盲猜，但需要有人力标注 |

## 五、技术栈速查

**长文档处理**

| 工具 | 用途 |
|:--|:--|
| Unstructured.io / LlamaParse | PDF/Word/PPT 结构提取，保留阅读顺序 |
| RAPTOR / LangChain Map-Reduce | 递归摘要树 + 多层级索引 |
| LongRAG / Multi-Vector Embedding | 大 chunk 检索，减少 chunk 总数 |

**大知识库检索**

| 工具 | 用途 |
|:--|:--|
| Milvus (HNSW/DiskANN) | 高维向量相似度搜索，亿级规模 |
| Elasticsearch (BM25) | 倒排索引精确匹配，互补语义检索 |
| bge-reranker-v2-m3 / Cohere Rerank v3 | 粗召回后精排 |
| Redis | 高频 query 的 top-k 结果缓存 |

**查询层**

| 工具 | 用途 |
|:--|:--|
| LLM Query Decomposition | 复杂问题拆多个子 query |
| LLM Router / Self-Query | 路由到不同知识库 + 自动提取元数据条件 |
| Reciprocal Rank Fusion | 多路召回结果排序融合 |

**评估层**

| 工具 | 用途 |
|:--|:--|
| RAGAS / DeepEval / TruLens | 检索精度 + 生成忠实度 + 答案相关性 |

---

> 总结：长文档靠"分层摘要 + 多粒度索引"解决，大知识库靠"粗筛 → 精排 → 融合"解决。两者叠加时不要一步到位，优先上 Reranker 和元数据过滤，这两个 ROI 最高。
