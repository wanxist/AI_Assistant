# RAG 生产经验与建议

## 1. 检索阶段

混合检索不是可选项，是必选项。你已经做了（向量 + BM25），保持。

Query Rewriting 比 HyDE 更实用。你现在用 HyDE（让 LLM 生成假答案去搜），但生产上更常用的是 Query Rewriting——把用户的模糊问题改写为精确的搜索 query。HyDE 多一次 LLM 调用，Query Rewriting 只需要规则或小模型。

Chunk 大小要按场景调。你现在 1024 字符 + 100 重叠——对制度文件 OK，但对合同、报表不合适。生产上通常提供 2-3 种 chunk 策略（细粒度 256 + 粗粒度 1024），检索时同时搜两种。

## 2. 生成阶段

永远给 LLM 编号的来源。你已经做了 [1] (文件名)，这是对的。生产级更进一步——来源可点击跳转到原文位置。

Temperature 设为 0。RAG 场景下不需要创意，要准确。你已经设了 0，保持。

限制 LLM 不要编造。Prompt 里已经写了"不要强行编造"，但 LLM 还是会编。最有效的方法是如果检索得分太低，直接告诉用户"没找到"，不调 LLM。你已经有得分阈值 0.35。

## 3. 评估与迭代

没有 eval 的 RAG 是盲飞。准备 20-30 个标准问题 + 预期答案，每次改 prompt/chunk 策略后跑一遍对比。最简单的 eval：看答案是否包含预期关键词。

## 4. 用户体验

显示"正在搜索"的状态。你已经做了思考过程动画，这很好。

答案太长用户不看。优先返回 3-5 条要点，然后是详细解释。Prompt 里加一句"先用要点回答，再展开"。

## 5. 运维

监控检索质量。如果大量查询返回"没有找到"，说明 chunk 策略或文档覆盖有问题。日志里已经有 RAG 查询记录。

文档更新策略。现在上传是覆盖的——同一文件 MD5 去重。生产上需要版本管理：文档 v1 → v2 时，旧向量自动过期。

## 对你项目最实用的三条

| 优先级 | 建议 | 理由 |
|--------|------|------|
| 高 | 准备 20 个 eval 问题 | 每次改 prompt 能立刻知道有没有变好 |
| 高 | 检索得分阈值调优 | 现在 0.35 是拍脑袋的，应该用 eval 跑出最佳值 |
| 中 | Chunk 大小可配置 | 不同文档类型用不同 chunk size |
| 中 | 来源可点击 | 答案里的 [1] 能跳转到文档详情 |
| 低 | 文档版本管理 | 同名文档更新时旧向量自动失效 |
